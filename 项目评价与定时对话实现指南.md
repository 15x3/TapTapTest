# 项目评价与定时对话实现指南

> 本文档包含两部分内容：对当前项目结构的评价，以及在当前架构下实现"到指定时间后自动触发对话"功能的完整指南。

---

## 第一部分：项目结构评价

### 1. 项目概况

这是一个基于 **Lua** 开发的手机界面模拟器，采用 **像素风格**，模拟了一个包含叮钉（类似钉钉）和微言（类似微信）的智能手机系统。项目采用 **事件驱动的对话系统**，数据存储在 **CSV** 文件中，便于策划编辑。

**核心特性：**
- 像素风格手机 UI（urhox-libs/UI 框架）
- 完整的对话事件状态机
- 虚拟时间系统（GameTime）
- 全局标签系统（TagManager）
- CSV 数据驱动（策划友好）
- 自动输入功能（"随意敲打键盘"）
- 关键词分支 + 阈值分支选择

---

### 2. 架构优点

**评价：** 职责划分明确，数据层、业务逻辑层、UI 层分离良好，符合单一职责原则。

---

#### ✅ 数据驱动（CSV）设计优秀

对话内容存储在 CSV 文件中，支持丰富的字段：

| 字段 | 说明 |
|------|------|
| `id` | 事件唯一标识 |
| `delay` | 延迟秒数 |
| `type` | 事件类型（message/typing/wait_input/choice/set_time/freeze_time） |
| `sender` | 发送者 |
| `text` | 消息内容 |
| `next` | 下一个事件 ID |
| `options` | 分支选项 |
| `tag` | 触发的全局标签 |
| `require_tag` | 前置标签条件 |
| `thresholds` | 阈值分支规则 |

**评价：** 策划无需修改代码即可调整对话流程，降低了开发-策划沟通成本。

---

#### ✅ 状态机设计健壮

`ChatEventManager` 实现了完整的状态机：

```
processing → delay → execute → processing / wait_input / wait_choice / done
```

支持以下状态：
- `processing`: 连续处理事件链
- `delay`: 等待延迟时间结束
- `wait_input`: 等待用户输入文本
- `wait_choice`: 等待用户选择分支
- `done`: 事件链执行完毕

**评价：** 状态转换逻辑清晰，易于扩展新的事件类型。

---

#### ✅ 虚拟时间系统设计优雅

`GameTime` 模块提供：
- 可自定义起始时间（默认周二 08:00）
- 时间冻结/解冻功能
- 时间跳转（`SetTime`, `SetWeekday`）
- 脏标记机制（用于 UI 刷新）

**评价：** 解耦了真实时间与游戏时间，支持对话中"时间跳跃"的叙事需求。

---

#### ✅ 全局标签系统灵活

`TagManager` 实现跨聊天的条件触发：
- 事件可写入标签（`tag` 字段）
- 事件可检查标签（`require_tag` 字段）
- 支持批量标签（`|` 分隔）

**评价：** 实现了剧情分支的"蝴蝶效应"，增加了叙事深度。

---

#### ✅ 自动输入功能创新

"随意敲打键盘"功能：
- 用户只需随机按键，系统自动逐字填充预设文本
- 支持多文本轮换（清空后切换下一条）
- 兼容中文输入法

**评价：** 降低了玩家输入负担，同时保持了互动感。

---

### 3. 架构缺点与改进建议

#### ⚠️ 缺少全局事件调度器

**当前问题：**
- 对话触发完全依赖用户主动打开聊天
- 无法实现"到指定时间自动触发对话"功能
- 没有全局的定时任务系统

**改进建议：**
（详见第二部分实现指南）

---

#### ⚠️ 数据层缺少运行时状态持久化

**当前问题：**
- 所有状态保存在内存中
- 应用重启后丢失所有进度（已读消息、标签状态等）

**改进建议：**
```lua
-- 建议添加 SaveDataManager.lua
local SaveDataManager = {
  savePath = "save_data.json",
  data = {
    tags = {},
    readChats = {},
    todoState = {},
  },
  function Save() -- 序列化到 JSON
  function Load() -- 从 JSON 反序列化
}
```

---

#### ⚠️ CSV 解析缺少错误处理

**当前问题：**
- `CSVParser.Load` 缺少对格式错误的处理
- 策划填写错误时可能导致崩溃

**改进建议：**
```lua
-- 在 CSVParser.lua 中添加
local function validateRow(row, lineNumber)
  if not row.id or row.id == "" then
    print(string.format("[警告] 第 %d 行缺少 id 字段", lineNumber))
    return false
  end
  -- 更多验证...
  return true
end
```

---

#### ⚠️ 缺少日志系统

**当前问题：**
- 调试困难，无法追踪状态转换
- 策划测试时难以定位问题

**改进建议：**
项目已有 `Utils/Logger.lua`，建议在关键路径添加日志：
```lua
-- ChatEventManager.lua
function Manager:Execute()
  Logger:debug("执行事件", self.index, event.type)
  -- ...
end
```

---

#### ⚠️ UI 层与业务逻辑耦合

**当前问题：**
- `ChatPage.lua` 直接创建 `ChatEventManager`
- UI 层承担了部分业务逻辑

**改进建议：**
引入中介者模式：
```lua
-- ChatController.lua
local ChatController = {
  function OpenChat(chatName)
    local events = DingtalkData.GetChatScenario(chatName)
    local manager = ChatEventManager.Create(events, callbacks)
    return manager
  end
}
```

## 第二部分：定时对话功能实现指南

### 功能需求

> **到指定时间后自动触发对话**
>
> 当虚拟时间到达预设时刻（如 14:30），自动弹出某个聊天的对话事件，无需用户主动打开。

---

### 方案设计

#### 方案对比

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **方案1：全局调度器** | 集中管理，易扩展 | 需新增模块 | ⭐⭐⭐⭐⭐ |
| 方案2：在 ChatEvent 添加时间触发字段 | 实现简单 | 分散管理，难维护 | ⭐⭐⭐ |
| 方案3：在 GameTime 添加回调列表 | 耦合度高 | 污染 GameTime 职责 | ⭐⭐ |

**推荐方案1：全局调度器**

---

### 实现步骤

#### 步骤 1：创建全局事件调度器

**文件：** `scripts/Utils/EventScheduler.lua`

```lua
-- ============================================================================
-- EventScheduler - 全局事件调度器
-- 功能: 在指定虚拟时间触发预设事件
-- 用法:
--   EventScheduler.Schedule({
--     hour = 14,
--     min = 30,
--     app = "dingtalk",
--     chatName = "张老师",
--     eventId = "event_id",  -- 可选，指定从哪个事件开始
--     once = true,           -- 可选，是否只触发一次
--   })
-- ============================================================================

local GameTime = require("Utils.GameTime")

local Scheduler = {}

---@class ScheduledEvent
---@field hour number 小时 (0-23)
---@field min number 分钟 (0-59)
---@field app string "dingtalk" | "wechat"
---@field chatName string 聊天名称
---@field eventId string|nil 可选，指定从哪个事件开始
---@field once boolean|nil 是否只触发一次
---@field triggered boolean|nil 是否已触发过

---@type ScheduledEvent[]
local scheduledEvents_ = {}

---@type table<string, boolean>
local triggeredToday_ = {}  -- 今日已触发的事件记录

--- 添加定时事件
---@param config ScheduledEvent
function Scheduler.Schedule(config)
  table.insert(scheduledEvents_, {
    hour = config.hour,
    min = config.min,
    app = config.app,
    chatName = config.chatName,
    eventId = config.eventId,
    once = config.once or false,
    triggered = false,
  })

  -- 按时间排序（早的在前）
  table.sort(scheduledEvents_, function(a, b)
    local timeA = a.hour * 60 + a.min
    local timeB = b.hour * 60 + b.min
    return timeA < timeB
  end)
end

--- 移除所有定时事件
function Scheduler.Clear()
  scheduledEvents_ = {}
  triggeredToday_ = {}
end

--- 获取当前应该触发的事件
---@return ScheduledEvent|nil
function Scheduler.CheckTriggers()
  local now = GameTime.Now()
  local currentMinutes = now.hour * 60 + now.min
  local todayKey = string.format("%04d-%02d-%02d", now.year, now.month, now.day)

  for _, event in ipairs(scheduledEvents_) do
    local eventMinutes = event.hour * 60 + event.min

    -- 检查时间是否到达
    if currentMinutes >= eventMinutes then
      -- 检查是否已触发过
      local eventKey = string.format("%s_%s_%s_%d:%d",
        event.app, event.chatName,
        event.eventId or "default",
        event.hour, event.min)

      if event.once and event.triggered then
        -- 标记为已触发，跳过
      elseif not triggeredToday_[eventKey] then
        -- 首次触发，标记并返回
        event.triggered = true
        triggeredToday_[eventKey] = true
        return event
      end
    end
  end

  return nil
end

--- 切换日期时清除当日触发记录
--- （应在主循环检测到日期变更时调用）
function Scheduler.ResetDaily()
  triggeredToday_ = {}
end

return Scheduler
```

**解决方案 B：使用全局事件总线**

创建 `scripts/Utils/EventBus.lua`：

```lua
local EventBus = {
    listeners = {},
}

function EventBus.On(eventName, callback)
    if not EventBus.listeners[eventName] then
        EventBus.listeners[eventName] = {}
    end
    table.insert(EventBus.listeners[eventName], callback)
end

function EventBus.Emit(eventName, ...)
    if EventBus.listeners[eventName] then
        for _, cb in ipairs(EventBus.listeners[eventName]) do
            cb(...)
        end
    end
end

return EventBus
```

在 `main.lua` 中：

```lua
EventBus.Emit("OpenChat", { app = "dingtalk", chatName = "张老师" })
```

在 `DingtalkApp.lua` 中：

```lua
EventBus.On("OpenChat", function(params)
    if params.app == "dingtalk" and params.chatName then
        OpenChatByName(params.chatName)
    end
end)
```

## 结语

当前项目架构设计优秀，模块化清晰，数据驱动设计降低了策划修改成本。通过添加全局事件调度器（`EventScheduler`），可以优雅地实现"到指定时间自动触发对话"功能，且不会破坏现有架构。

**关键要点：**
1. 使用全局单例管理所有定时事件
2. 在主循环的 `HandleUpdate` 中检查触发条件
3. 通过事件总线或参数传递实现应用层解耦
4. 数据源可灵活配置（CSV/代码/数据库）
