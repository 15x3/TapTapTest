# 一日班主任 —— 程序员需求文档

> **版本**: v1.0
> **日期**: 2026-03-27
> **状态**: 初版，待开发评审

---

## 1. 项目概述

### 1.1 游戏简介

**一日班主任** 是一款模拟班主任课间处理信息的生存游戏。玩家在 10 分钟的课间休息内，从多个聊天群的海量信息流中筛选有效信息，通过转发、回复、发公告三种操作正确处理，游戏通过群内人物的自然反应提供即时反馈。

### 1.2 参考作品

- **《S4U：都市朋克2011与爱的重拳》** —— 敲键盘驱动文字、多线程聊天的操作手感
- 现有 demo（班主任晨间大作战）—— 手机模拟器 UI 框架、CSV 数据驱动架构

### 1.3 现有代码基础

当前项目约 8000 行 Lua 代码，已实现：
- 手机模拟器外壳（叮叮/微言/课表三个 App）
- CSV 数据驱动的对话系统（ChatEventManager）
- 关键词检索 + 情感分析模块
- 虚拟时间系统（GameTime）
- 定时事件调度器（EventScheduler）
- S4U 式"敲键盘自动填字"机制

**需要改造/新增的部分见第 3-6 节。**

## 2. 核心游戏循环

```
关卡简报（情境上下文，1-2 句话）
        ↓
10 分钟实时信息流（信息量设计为不可能全部处理完）
        ↓
玩家操作：转发 / 回复 / 发公告
        ↓
人物自然反应（即时反馈）
        ↓
信息处理报告（关卡结算）
```

## 3. 三个核心操作（详细交互流程）

### 3.1 转发消息

**流程**：

```
玩家在聊天详情页看到一条消息
    ↓
长按该消息（触发长按事件，约 0.5 秒）
    ↓
弹出操作菜单（浮层）：
    ┌──────────┐
    │  转发     │
    │  复制     │
    │  取消     │
    └──────────┘
    ↓ 点击"转发"
弹出群/联系人选择列表：
    ┌──────────────────┐
    │ 选择转发目标       │
    │                    │
    │ 📋 七年三班家长群   │
    │ 📋 七年三班班级群   │
    │ 📋 年级工作群      │
    │ 👤 张主任          │
    │ ...               │
    │                    │
    │    [取消]          │
    └──────────────────┘
    ↓ 点击目标群
确认弹窗："转发到 [七年三班家长群]？"
    ↓ 确认
消息出现在目标群的聊天记录中（以"转发消息"样式显示）
目标群的人物开始反应（见第 5 节反馈机制）
```

**技术要点**：
- 长按检测：在聊天气泡组件上监听按下时间，超过 500ms 触发菜单
- 转发目标列表：从当前关卡配置中读取可用的群/联系人列表
- 转发记录：需要记录"哪条消息被转发到了哪里"，用于结算判定
- 错误转发：如果转发到了错误的目标，也要正常执行并触发负面反馈

**数据结构建议**：

```lua
-- 转发记录
ForwardRecord = {
    source_chat = "年级工作群",       -- 来源群
    message_id = "msg_042",           -- 原消息 ID
    target_chat = "七年三班家长群",    -- 转发目标
    timestamp = 185.0,                -- 游戏内时间（秒）
    is_correct = true,                -- 是否为正确的转发（结算时判定）
}
```

### 3.2 回复消息

**流程**：

```
玩家在聊天详情页点击底部输入框
    ↓
输入框获得焦点，显示光标
    ↓
玩家敲击键盘任意键（S4U 式）
    ↓
系统逐字符填入预设回复文本（CSV 配置）
    ↓
文本填完后，玩家点击"发送"按钮（或按回车）
    ↓
消息以玩家身份出现在聊天记录中
对方做出反应（见第 5 节反馈机制）
```

**技术要点**：
- 复用现有 ChatEventManager 中的 `auto_fill` 机制
- 预设回复内容在 CSV 中配置
- 每次按键填入 1-2 个字符，模拟真实打字节奏
- 如果有多个可选回复（分支），根据上下文或玩家之前的操作决定填入哪个

### 3.3 发布公告

**流程**：

```
玩家进入目标群（如"七年三班班级群"）
    ↓
点击输入框上方的"公告"按钮（或群功能入口）
    ↓
弹出公告编辑界面：
    ┌──────────────────────┐
    │ 发布班级公告           │
    │                        │
    │ ┌────────────────────┐ │
    │ │                    │ │
    │ │ （自由输入区域）     │ │
    │ │                    │ │
    │ │                    │ │
    │ └────────────────────┘ │
    │                        │
    │  [取消]     [发布]     │
    └──────────────────────┘
    ↓
玩家用真实键盘输入公告内容（非 S4U 式，完全自由输入）
    ↓
点击"发布"
    ↓
系统对输入内容进行关键词检测（见下方）
    ↓
公告以特殊样式出现在群聊天记录中
群内成员根据公告质量做出反应
```

**关键词检测逻辑**：

```lua
-- 每个关卡定义需要提及的关键信息组
AnnouncementKeywords = {
    {
        group = "调课通知",
        keywords = {"调课", "换课", "课程调整"},  -- 命中任一即可
        detail_keywords = {"周三", "下午"},        -- 额外加分项
        weight = 1,                                -- 重要程度
    },
    {
        group = "体检通知",
        keywords = {"体检", "体检表"},
        detail_keywords = {"下午", "交"},
        weight = 1,
    },
}

-- 检测结果
AnnouncementResult = {
    mentioned = {"调课通知"},              -- 提到了哪些信息组
    missed = {"体检通知"},                 -- 遗漏了哪些
    detail_score = 1,                      -- 额外细节命中数
    total_groups = 2,                      -- 总共应提及的信息组数
    mentioned_count = 1,                   -- 实际提及数
}
```

**技术要点**：
- 输入框需要支持**真实文本输入**（非 S4U 的 auto_fill 模式）
- 关键词匹配使用现有的关键词检索功能，扩展为"信息组"粒度
- 输入法支持：需要处理中文输入法的 composition 事件
- 公告内容和检测结果需要记录，用于结算

## 4. 信息流系统

### 4.1 消息调度

每个关卡的消息流由 CSV 数据驱动，按时间轴调度：

```csv
time,chat,sender,type,content,priority,forward_target,keywords
30,年级工作群,年级主任,message,各位班主任：周三下午第六七节调课为自习，请通知到每位家长。,high,七年三班家长群,"调课,周三,下午"
45,七年三班家长群,李妈妈,message,老师早上好呀[太阳],noise,,
60,七年三班家长群,王爸爸,message,[图片]早安,noise,,
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `time` | int | 关卡开始后的秒数（0-600） |
| `chat` | string | 消息出现在哪个群/私聊 |
| `sender` | string | 发送者名称 |
| `type` | enum | `message` / `image` / `forward` / `system` |
| `content` | string | 消息内容 |
| `priority` | enum | `high`（必须处理）/ `low`（可处理可不处理）/ `noise`（噪音） |
| `forward_target` | string | 正确的转发目标（为空表示不需要转发） |
| `keywords` | string | 关联的关键词（逗号分隔），用于公告检测 |

### 4.2 消息密度设计

10 分钟（600 秒）关卡的消息分布建议：

| 类别 | 数量占比 | 说明 |
|------|---------|------|
| 高优先级（必须处理） | ~25% | 需要转发/回复的重要信息 |
| 低优先级（可选处理） | ~15% | 回复了更好，不回也行 |
| 噪音（应忽略） | ~40% | 早安图、闲聊、不相关讨论 |
| 反馈消息（系统生成） | ~20% | 根据玩家操作动态产生 |

### 4.3 未读提示与通知

- 每个群维护独立的未读计数
- 新消息到达时，如果玩家不在该群，群列表上显示未读红点/数字
- 高优先级消息到达时，顶部弹出通知横幅（已有实现）

## 5. 反馈机制

### 5.1 即时反馈（游戏过程中）

反馈通过群内 NPC 自动发送的消息实现，**不使用积分、评分等抽象数值**。

**正确操作的反馈**：

| 场景 | 反馈消息 |
|------|---------|
| 正确转发通知到家长群 | 家长A："收到，谢谢老师" / 家长B："好的" |
| 及时回复家长私聊 | 家长："谢谢老师，麻烦您了" |
| 及时回复学生求助 | 学生："谢谢老师！" |

**错误操作的反馈**：

| 场景 | 反馈消息 |
|------|---------|
| 通知转发到了错误的群 | 群成员："老师这是发错了吧？" |
| 长时间未处理重要消息 | 发送者追问 / 领导私聊询问："XX 的事处理了吗？" |
| 把内部消息转到家长群 | 群内短暂沉默 → 家长："这个是发给我们看的吗？" |

**技术实现**：
- 每条高优先级消息关联一组"反馈触发器"
- 触发器有两类：
  - **正向触发**：玩家正确操作后 N 秒触发
  - **超时触发**：玩家未操作超过 M 秒触发（催促/追问）
- 反馈消息也是通过消息调度系统发出，和普通消息走同一套渲染逻辑

```lua
-- 反馈触发器数据结构
FeedbackTrigger = {
    source_message_id = "msg_042",
    -- 正确操作时的反馈
    on_correct = {
        delay = 5,                          -- 正确操作后 5 秒触发
        chat = "七年三班家长群",
        responses = {
            {sender = "李妈妈", content = "收到，谢谢老师"},
            {sender = "王爸爸", content = "好的"},
        }
    },
    -- 超时未处理的反馈
    on_timeout = {
        timeout = 120,                      -- 120 秒未处理
        chat = "年级工作群",
        responses = {
            {sender = "年级主任", content = "@你 调课通知转发了吗？"},
        }
    },
    -- 错误操作的反馈
    on_wrong_target = {
        chat = "$target",                   -- 动态：出现在被错误转发到的群
        responses = {
            {sender = "$random_member", content = "老师这是发错了吧？"},
        }
    },
}
```

### 5.2 结算反馈（关卡结束后）

关卡结束后显示 **信息处理报告**，列出具体条目：

```
┌─────────────────────────────────────┐
│         信息处理报告                  │
│                                      │
│  ✅ 已处理                           │
│  ├─ 调课通知 → 已转发至家长群         │
│  ├─ 家长咨询（眼镜坏了）→ 已回复      │
│  └─ 张主任询问体检表 → 已回复         │
│                                      │
│  ❌ 遗漏                             │
│  ├─ 体检注意事项 → 未转发至家长群     │
│  └─ 学生反映打架 → 未回复            │
│                                      │
│  ⚠️ 公告质量                         │
│  ├─ ✅ 提到了：调课                   │
│  └─ ❌ 遗漏了：体检                   │
│                                      │
│          [再来一次]  [下一关]          │
└─────────────────────────────────────┘
```

**技术要点**：
- 结算数据来源：转发记录 + 回复记录 + 公告关键词检测结果 + 超时触发器是否被触发
- 不显示分数、星级、百分比等数值化评价
- 只展示事实："你做了什么，漏了什么"

## 6. 关卡系统

### 6.1 关卡配置

每个关卡由以下数据文件定义：

```
data/levels/level_01/
├── config.csv          # 关卡基础配置（时长、简报文本）
├── messages.csv        # 消息时间轴（见 4.1 节格式）
├── chats.csv           # 本关可用的群/联系人列表
├── contacts.csv        # 本关出现的 NPC 列表
├── feedbacks.csv       # 反馈触发器配置
└── announcement.csv    # 公告关键词配置
```

### 6.2 关卡流程（程序控制）

```lua
-- 伪代码
function StartLevel(levelId)
    -- 1. 加载关卡数据
    LoadLevelData(levelId)

    -- 2. 显示情境简报（全屏遮罩 + 文字）
    ShowBriefing(levelConfig.briefing_text)

    -- 3. 玩家点击"开始"后
    StartTimer(600)  -- 10 分钟倒计时
    StartMessageScheduler()  -- 开始按时间轴推送消息

    -- 4. 游戏进行中...
    -- 玩家操作 → 记录到 ActionLog
    -- 触发器检测 → 推送反馈消息

    -- 5. 时间到
    StopMessageScheduler()
    ShowAnnouncementPrompt()  -- 如果本关需要发公告

    -- 6. 结算
    local report = GenerateReport(ActionLog, FeedbackTriggers, AnnouncementResult)
    ShowReport(report)
end
```

### 6.3 计时器 UI

- 手机状态栏区域显示剩余时间（融入手机界面，不要用突兀的 HUD）
- 最后 60 秒时间数字变为警告色
- 时间结束时有提示音效和短暂的屏幕效果

## 8. 数据格式规范

### 8.1 与现有 CSV 格式的关系

现有 `chat_scenarios.csv` 格式需要扩展以支持新字段（`priority`、`forward_target`、`keywords`）。建议**新建关卡专用 CSV**，保持现有格式不变，避免破坏已有功能。

### 8.2 建议的迁移策略

1. 在 `data/` 下新建 `levels/` 目录，按关卡组织数据
2. 新的消息调度器（LevelMessageScheduler）独立于现有 ChatEventManager
3. 现有的 ChatEventManager 中的关键词检索、情感分析模块可复用
4. 现有的 S4U 式自动填字机制可直接复用于"回复"操作

## 9. 开发优先级建议

### Phase 1：核心机制验证（最小可玩版本）

1. 关卡计时器（10 分钟倒计时）
2. 基于时间轴的消息调度（替代现有的事件触发式）
3. 转发操作完整流程（长按 → 菜单 → 选目标 → 确认）
4. 信息处理报告（结算页面）
5. 第一关的完整数据

### Phase 2：丰富体验

6. 发公告操作 + 关键词检测
7. 即时反馈系统（NPC 自动回复）
8. 噪音消息系统
9. 关卡简报界面

### Phase 3：打磨

10. 倒计时视觉效果（最后 60 秒警告）
11. 错误转发的反馈
12. 超时催促机制
13. 音效
