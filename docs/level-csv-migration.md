# 关卡模式 CSV 迁移说明

> 本文档面向文案同学，说明关卡模式下消息系统的调整，以及 `messages.csv` 的编写规范。

---

## 一、发生了什么变化

### 之前的问题

项目中原有两套消息系统**同时运行**：

| 系统 | 数据文件 | 驱动方式 |
|------|---------|---------|
| **场景事件系统** | `data/chat_scenarios.csv` | 打开聊天时按事件链自动播放 |
| **关卡消息系统** | `data/levels/level1/messages.csv` | 按关卡倒计时时间轴投放 |

`chat_scenarios.csv` 中标记为 `*`（通配）的事件会在**所有聊天**中播放。这导致玩家打开任何聊天（如"周老师"）时，除了看到关卡消息，还会同时看到大量不相关的场景剧情（家长互动、转发选择题、心理普查等），造成严重混乱，包括：

- 时间被意外冻结（场景事件中的 `freeze_time` 触发）
- 大量不属于该聊天的消息涌入
- 交互选择题出现在不该出现的位置

### 现在的调整

**关卡模式下，`chat_scenarios.csv` 的场景事件不再加载。**

关卡内所有消息统一通过 `messages.csv` 按时间轴投放。非关卡模式（如待机界面、未来的叙事模式）仍然可以使用 `chat_scenarios.csv`。

---

## 二、对文案工作的影响

### 需要关注

1. **关卡内的所有消息、系统提示都必须写在 `messages.csv` 中**，不能依赖 `chat_scenarios.csv`
2. 原来写在 `chat_scenarios.csv` 中、且关卡内需要展示的内容，需要迁移到 `messages.csv`

### 不受影响

1. `chat_scenarios.csv` 文件本身保留，非关卡模式下仍然有效
2. 新手教程如果放在场景事件中，需要另行讨论是否也迁移到关卡消息系统

---

## 三、messages.csv 完整字段说明

文件路径：`data/levels/{关卡ID}/messages.csv`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `time_offset` | 数字 | 是 | 消息投放时间（秒），从关卡开始计算。如 `30` 表示关卡开始后第 30 秒投放 |
| `app` | 文本 | 是 | 目标 App：`dingtalk`（叮叮）或 `wechat`（微言） |
| `chat` | 文本 | 是 | 目标聊天名称，必须与 `chats.csv` 中的名称一致 |
| `sender` | 文本 | 是 | 发送者名称，显示在气泡上 |
| `type` | 文本 | 是 | 消息类型，见下方详解 |
| `content` | 文本 | 是 | 消息正文内容 |
| `priority` | 文本 | 否 | 优先级标记，见下方详解 |
| `forward_target` | 文本 | 否 | 正确的转发目标群，多个用 `\|` 分隔 |
| `keywords` | 文本 | 否 | 回复关键词，用 `\|` 分隔，匹配任一即算正确回复 |
| `reply_hint` | 文本 | 否 | 自动填充的回复文本（"随意敲打键盘"功能），留空则玩家需手动输入 |
| `chain_id` | 文本 | 否 | 信息链 ID，同一链的消息共享此 ID（如 `A`），用于结算分组 |
| `chain_name` | 文本 | 否 | 信息链名称，结算时显示（如"调课通知"） |

### type 字段详解

| 值 | 说明 | 玩家需要做什么 |
|----|------|--------------|
| `message` | 普通消息 | 阅读即可 |
| `wait_reply` | 需要玩家回复的消息 | 在输入框回复，系统通过 `keywords` 判断是否正确 |
| `system` | 系统提示 | 以灰色居中样式显示，无气泡，用于引导玩家 |

### priority 字段详解

| 值 | 说明 | 结算影响 |
|----|------|---------|
| `important` | 重要消息 | 需要转发/回复，计入结算统计 |
| `noise` | 噪音消息 | 干扰项，不需要处理 |
| `normal` | 一般消息 | 默认值，系统提示等 |

## 四、编写示例

### 示例 1：普通群聊灌水（噪音）

```csv
45,dingtalk,年级工作群,周老师,message,又调课，这学期第几次了,noise,,,,,
60,dingtalk,年级工作群,陈老师,message,第三次了吧，习惯就好,noise,,,,,
```

### 示例 2：需要转发的重要通知

```csv
30,dingtalk,年级工作群,刘主任,message,@所有人 通知：本周三下午调课为自习课，请各班主任通知到每位家长。,important,七(3)班家长群|七年三班家校群,,,A,调课通知
```

- `priority=important`：标记为重要，结算时检查
- `forward_target`：正确转发目标是"七(3)班家长群"和"七年三班家校群"
- `chain_id=A`：信息链标识，同一个事件链用同一个 ID
- `chain_name=调课通知`：结算时显示的链名称

### 示例 3：需要玩家回复的消息

```csv
180,dingtalk,张主任,张主任,wait_reply,老师辛苦了，上周体检表各班收齐了吗？麻烦今天下午放学前交到教务处哈~,important,,,收到|好的|没问题|交|齐了|在收,B,体检表催收
```

- `type=wait_reply`：等待玩家回复
- `keywords`：玩家输入包含"收到"、"好的"、"没问题"等任一关键词即算正确
- `reply_hint`（当前为空）：若填写如"收到，我会尽快整理提交"，则玩家随意敲键盘时会自动填充此文本

### 示例 4：系统提示引导

```csv
540,dingtalk,七年三班班级群,系统,system,还剩1分钟，别忘了在班级群发布公告,normal,,,,,
```

- `type=system`：显示为灰色居中提示条，不带头像和气泡
- 用于在关键时刻给玩家提示

## 五、信息链（chain）机制

同一个事件的相关消息使用相同的 `chain_id`，结算时会按链分组统计。

```
chain_id = A  →  "调课通知"
  ├── [30s]  刘主任发通知（需要转发）
  └── 玩家转发到家长群 → 结算记录"已转发"

chain_id = B  →  "体检表催收"
  ├── [180s] 张主任催收（需要回复）
  └── 玩家回复 → 结算记录"已回复"

chain_id = C  →  "体检注意事项"
  ├── [240s] 刘主任发通知（需要转发）
  └── 玩家转发到家长群 → 结算记录"已转发"
```

`reply_hint` 填写后，玩家在输入框随意按键时会自动逐字填充该文本，降低打字负担。留空则需要玩家完全手动输入。

## 七、时间轴规划建议

关卡总时长 600 秒（10 分钟），建议消息分布遵循以下节奏：

```
0-60s     开场铺垫：周老师提示 + 家长群灌水（noise）
30-120s   第一波重要：调课通知（chain A）
120-200s  第二波重要：家长求助（chain D） + 体检催收（chain B）
240-300s  第三波重要：体检通知（chain C）+ 学生事件（chain E）
300-480s  持续噪音 + 零散消息
540s      系统提示：还剩1分钟
600s      关卡结束 → 进入结算
```

每条消息的 `time_offset` 可以自由调整，系统会严格按秒数投放。
