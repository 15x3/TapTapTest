# 关卡数据编写指南

> 面向文案团队的关卡数据填写手册
> 每个关卡是一个文件夹，包含 5 个 CSV 文件

---

## 目录结构

```
scripts/data/levels/
  level1/              ← 第 1 关
    config.csv         ← 关卡基本信息
    chats.csv          ← 本关涉及的聊天列表
    messages.csv       ← 消息时间轴（核心）
    feedbacks.csv      ← NPC 反馈规则
    announcement.csv   ← 公告检查点
  level2/              ← 第 2 关（同样结构）
    ...
```

新建关卡：复制 `level1/` 文件夹，重命名为 `level2/`，修改内容即可。

## 1. config.csv — 关卡基本信息

只需 **1 行数据**。

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `name` | 文本 | 关卡名称 | 课间十分钟 |
| `duration` | 数字 | 关卡时长（秒） | 600 |
| `start_hour` | 数字 | 游戏内起始小时 | 9 |
| `start_min` | 数字 | 游戏内起始分钟 | 40 |
| `briefing` | 文本 | 任务简报（开始前展示给玩家） | 课间休息时间到了... |
| `objective` | 文本 | 目标提示（一句话） | 在10分钟内妥善处理所有重要消息 |

**示例**：
```csv
name,duration,start_hour,start_min,briefing,objective
课间十分钟,600,9,40,课间休息时间到了。作为新上任的班主任...,在10分钟内妥善处理所有重要消息
```

## 2. chats.csv — 聊天列表

定义本关会出现的所有聊天窗口。

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `name` | 文本 | 聊天名称（必须与 messages.csv 中的 `chat` 完全一致） | 教务处通知群 |
| `app` | 枚举 | 所属 App：`dingtalk`（叮叮） 或 `wechat`（微言） | dingtalk |
| `icon_color` | 文本 | 头像背景色：blue/purple/green/cyan/pink/gray/orange/red | blue |
| `icon_text` | 文本 | 头像显示文字（1-2 字） | 教务 |
| `is_forward_target` | 枚举 | 是否可被选为转发目标：`yes` 或 `no` | yes |

**注意**：
- `is_forward_target = yes` 的聊天会出现在转发目标选择列表中
- 通常"家长群"、"班级群"等需要接收转发的群设为 `yes`
- "教务处通知群"、"年级主任群"等消息来源群设为 `no`
- 个人聊天（如"李妈妈"）通常设为 `no`

**示例**：
```csv
name,app,icon_color,icon_text,is_forward_target
教务处通知群,dingtalk,blue,教务,no
年级主任群,dingtalk,purple,年级,no
三年级一班家长群,dingtalk,green,家长群,yes
李妈妈,wechat,pink,李,no
小明(班长),wechat,orange,明,no
```

## 3. messages.csv — 消息时间轴（核心文件）

定义关卡中何时、由谁、在哪个聊天中发送什么消息。这是关卡剧情的主体。

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `time_offset` | 数字 | 消息出现时间（关卡开始后的秒数） | 30 |
| `app` | 枚举 | `dingtalk` 或 `wechat`；系统消息用 `system` | dingtalk |
| `chat` | 文本 | 聊天名称（必须与 chats.csv 中的 `name` 一致） | 教务处通知群 |
| `sender` | 文本 | 发送者名称 | 刘主任 |
| `type` | 枚举 | 消息类型（见下表） | message |
| `content` | 文本 | 消息正文 | 下周二三年级调课通知已出 |
| `priority` | 枚举 | 优先级：`important`/`normal`/`noise` | important |
| `forward_target` | 文本 | 需要转发到的目标聊天名称（可为空） | 三年级一班家长群 |
| `keywords` | 文本 | 保留字段（目前未使用，可为空） | |
| `reply_hint` | 文本 | 回复关键词，用 `\|` 分隔（仅 wait_reply 有效） | 收到\|好的\|明白 |
| `chain_id` | 文本 | 信息链 ID（用于结算分组，相同 ID 归为一组） | A |
| `chain_name` | 文本 | 信息链显示名称（结算报告中显示） | 调课通知转发 |

### 消息类型 (type)

| 类型 | 说明 | 玩家需要做什么 |
|------|------|--------------|
| `message` | 普通消息 | 如有 `forward_target`，需要长按转发 |
| `wait_reply` | 等待回复的消息 | 玩家需在对话框输入包含关键词的回复 |
| `system` | 系统提示（不显示在聊天中） | 无操作（用于公告检查点等内部事件） |

### 优先级 (priority)

| 优先级 | 说明 |
|--------|------|
| `important` | 重要消息，需要玩家操作（转发/回复） |
| `normal` | 普通消息，可能需要也可能不需要操作 |
| `noise` | 干扰消息，不需要操作（增加难度） |

### 转发目标 (forward_target)

- 留空 = 此消息不需要转发
- 填写一个聊天名 = 必须转发到该聊天
- 用 `|` 分隔多个 = 转发到任一个都算正确，如 `三年级一班家长群|家委会群`

### 回复关键词 (reply_hint)

仅在 `type = wait_reply` 时有效。用 `|` 分隔多个关键词，玩家回复中包含**任意一个**即算正确。

例如 `周五|这周五|本周五|交到` 表示玩家输入"周五交到办公室"会匹配成功。

### 信息链 (chain_id / chain_name)

同一个事件的多条相关消息用**相同的 chain_id** 标记。结算时会按信息链分组显示。

- `chain_id`：内部标识，用英文/数字，如 `A`、`B`、`event_1`
- `chain_name`：玩家看到的名称，用中文，如"调课通知转发"
- 不填 chain_id 的消息不会出现在结算报告中（如纯干扰消息）

**示例**：
```csv
time_offset,app,chat,sender,type,content,priority,forward_target,keywords,reply_hint,chain_id,chain_name
5,dingtalk,教务处通知群,刘主任,message,@所有班主任 下周二调课通知已出,important,三年级一班家长群,,,A,调课通知转发
60,wechat,李妈妈,李妈妈,message,老师您好！体检表我填好了,normal,,,收到|好的,B,体检表催收
120,wechat,李妈妈,李妈妈,wait_reply,请问体检表什么时候交呀？,normal,,,周五|本周五,B,体检表催收
180,wechat,王爸爸,王爸爸,message,老师好,noise,,,,,
480,system,system,system,system,公告检查点,normal,,,,,
```

## 4. feedbacks.csv — NPC 反馈规则

定义玩家操作后 NPC 的自动回应消息。

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `trigger` | 文本 | 触发条件（格式见下） | on_correct:forward:教务处通知群 |
| `source_chat` | 文本 | 关联的源聊天名称 | 教务处通知群 |
| `delay` | 数字 | 延迟时间（秒，含义因触发类型不同） | 15 |
| `app` | 枚举 | 反馈消息投递的 App | dingtalk |
| `chat` | 文本 | 反馈消息投递的聊天 | 教务处通知群 |
| `sender` | 文本 | 反馈消息的发送者 | 刘主任 |
| `content` | 文本 | 反馈消息的内容 | 收到，谢谢配合。 |

### 触发条件格式 (trigger)

```
on_correct:forward:聊天名     ← 玩家正确转发了该聊天的消息后触发
on_timeout:forward:聊天名     ← 玩家超时未转发该聊天的消息后触发
on_wrong_target:forward:聊天名 ← 玩家把该聊天的消息转发到错误目标后触发
on_correct:reply:聊天名       ← 玩家正确回复了该聊天后触发
on_timeout:reply:聊天名       ← 玩家超时未回复该聊天后触发
```

### delay 含义

| 触发类型 | delay 含义 |
|---------|-----------|
| `on_correct` | 正确操作后过多少秒发送反馈（模拟 NPC 反应时间） |
| `on_timeout` | 消息投递后过多少秒未操作则触发（超时窗口） |
| `on_wrong_target` | 错误操作后过多少秒发送反馈（通常设为 0 或很小） |

**设计建议**：
- `on_correct` 的 delay 通常 3~15 秒（NPC 看到后的回复延迟）
- `on_timeout` 的 delay 通常 120~300 秒（给玩家的操作窗口）
- 回复类型的 timeout 可以比转发类型短（回复更紧迫）

**示例**：
```csv
trigger,source_chat,delay,app,chat,sender,content
on_correct:forward:教务处通知群,教务处通知群,15,dingtalk,教务处通知群,刘主任,收到，谢谢配合。
on_timeout:forward:教务处通知群,教务处通知群,300,dingtalk,教务处通知群,刘主任,请尽快转发调课通知，家长们在等。
on_correct:reply:李妈妈,李妈妈,5,wechat,李妈妈,李妈妈,好的谢谢老师！那我周五让小李带过来。
on_timeout:reply:李妈妈,李妈妈,180,wechat,李妈妈,李妈妈,老师？您在忙吗？
on_correct:reply:小明(班长),小明(班长),3,wechat,小明(班长),小明,好的老师！我马上陪他去医务室！
on_timeout:reply:小明(班长),小明(班长),120,wechat,小明(班长),小明,老师？小张好像越来越不舒服了...
```

## 5. announcement.csv — 公告检查点

定义在某个时间点检查玩家是否已发布包含指定关键词的公告。

| 列名 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `check_time_offset` | 数字 | 检查时间（关卡开始后的秒数） | 480 |
| `required_keywords` | 文本 | 必须包含的关键词，用 `\|` 分隔 | 体检\|注意\|空腹 |

**检查逻辑**：到达指定时间时，系统检查玩家已发布的所有公告中是否有包含**全部**关键词的公告。

**注意**：关键词是"全部命中"逻辑（AND），而非"任一命中"。如果公告中只包含"体检"和"注意"但没有"空腹"，则检查不通过。

**示例**：
```csv
check_time_offset,required_keywords
480,体检|注意|空腹|禁食|禁水
```

> 上面的例子要求：在关卡开始 480 秒（8 分钟）时，玩家须已发布一条同时包含"体检"、"注意"、"空腹"、"禁食"、"禁水"这些关键词的公告。

## 设计技巧

### 节奏控制

- 关卡开头（0~60秒）：投放 1-2 条重要消息，让玩家热身
- 中段（60~300秒）：主要剧情展开，混合重要消息和干扰消息
- 后段（300~500秒）：催促未完成的操作（通过 feedbacks 的 timeout 实现）
- 结尾（480+秒）：公告检查点，收尾

### 难度调节

| 手段 | 低难度 | 高难度 |
|------|--------|--------|
| 消息间隔 | 30~60秒 | 5~15秒 |
| 干扰消息比例 | 少（20%） | 多（50%+） |
| 超时窗口 | 长（300秒） | 短（60秒） |
| 转发目标数 | 少（1个） | 多（3+个） |
| 回复关键词 | 宽泛（多个同义词） | 精确（少量关键词） |

### 信息链设计

一个好的信息链通常包含：
1. **引入消息**：NPC 发出初始信息（type=message）
2. **操作触发**：需要玩家转发或回复的消息
3. **催促提醒**：同一链中稍后的催促消息（增加紧迫感）

```
chain A "调课通知":
  t=5   刘主任发通知     → 需要转发
  t=360 刘主任再次催促   → 增加紧迫感
  反馈: 正确转发后刘主任说"收到谢谢"
  反馈: 超时未转发刘主任说"请尽快转发"
```

## 常见问题

**Q: 聊天名称必须完全一致吗？**
A: 是的。`chats.csv` 的 `name`、`messages.csv` 的 `chat`、`feedbacks.csv` 的 `source_chat` 和 trigger 中的聊天名，必须**完全一致**（包括括号、空格等）。

**Q: 消息可以同时需要转发和需要回复吗？**
A: 不可以。一条消息只能是 `message`（可能需要转发）或 `wait_reply`（需要回复）。如果一个聊天既有转发需求又有回复需求，请拆成多条消息。

**Q: forward_target 可以填多个目标吗？**
A: 可以，用 `|` 分隔。玩家转发到其中任一个都算正确。

**Q: 公告关键词是"全部命中"还是"任一命中"？**
A: 全部命中（AND 逻辑）。玩家发布的公告必须同时包含所有关键词。

**Q: 如何测试关卡？**
A: 在 `main.lua` 顶部的 `levelMode_` 变量设为 `true`，构建后即可预览。修改 CSV 数据后重新构建即可看到变化。
